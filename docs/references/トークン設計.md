# トークン設計（現行実装準拠）

この文書は `src/tenhou_tokenizer/engine.py` の現在の挙動に合わせた仕様書です。

## 1. 目的

- 天鳳牌譜に含まれない「選べたが選ばなかった行動」をトークン化する。
- 学習対象を「局面遷移 + 意思決定」に寄せる。
- 大規模ストリーミング処理に耐える簡潔な語彙を使う。

## 2. スコープ

- 対象イベント: `qipai`, `zimo`, `gangzimo`, `dapai`, `fulou`, `gang`, `kaigang`, `hule`, `pingju`
- 4人麻雀（四麻）前提
- `pymahjong` によるシャンテン/和了判定を利用

非スコープ（現時点）:

- 三麻（抜きドラ `bao` など）
- `hule` の役・符・翻数・和了形詳細の出力
- `rank_*` の明示トークン

## 3. 出力フォーマット（CLI）

`tokenize_tenhou.py` の1レコードは次の JSON:

```json
{
  "source_zip": ".../data2023.zip",
  "game_id": "...json",
  "tokens": ["game_start", "...", "game_end"]
}
```

## 4. トークン列の基本構造

1. 対局開始: `game_start`
2. 各局（round）を牌譜順に処理
3. 対局終了: `game_end`

局開始時は必ず `round_start` が出る。`round_seq_{n}` はゲーム内0始まり連番。

## 5. イベント別トークン仕様

### 5.1 qipai（配牌）

- `round_start`
- `round_seq_{n}`
- `bakaze_{zhuangfeng}`
- `kyoku_{jushu}`
- `honba_{changbang}`
- `riichi_sticks_{lizhibang}`
- `dora_{tile}`
- `score_{seat}_{value}`（4人分）
- `haipai_{seat}_{tile}`（初期手牌ぶん）

### 5.2 zimo / gangzimo（自摸）

- 通常自摸: `draw_{seat}_{tile}`
- 槓自摸: `gang_draw_{seat}_{tile}`
- 直後に自己行動候補を列挙:
  - `opt_self_{seat}_tsumo`
  - `opt_self_{seat}_riichi`
  - `opt_self_{seat}_ankan`
  - `opt_self_{seat}_kakan`

### 5.3 自己行動の確定（draw後）

自己候補は次の自己イベントで確定する:

- 採用: `take_self_{seat}_{action}`
- 不採用: `pass_self_{seat}_{action}`

確定契機:

- `dapai`（立直宣言を含む）
- `gang`（暗槓/加槓）
- `hule`（ツモ）
- `pingju`
- `kaigang` は自己決定の継続イベントとして扱い、ここでは確定しない

### 5.4 dapai（打牌）

- `discard_{seat}_{tile}`
- ツモ切りは `discard_{seat}_{tile}_tsumogiri`
- 立直宣言打牌なら追加で `riichi_{seat}`

### 5.5 打牌への副露/ロン反応候補

打牌後、他家の反応候補を列挙:

- `opt_react_{seat}_ron`
- `opt_react_{seat}_chi`
- `opt_react_{seat}_pon`
- `opt_react_{seat}_minkan`

確定時:

- 採用: `take_react_{seat}_{action}`
- 不採用: `pass_react_{seat}_{action}`

補足:

- ロン見逃し時はフリテン状態（通常/立直後）を内部更新し、以降の候補計算に反映。

### 5.6 fulou（チー/ポン/大明槓）

- `call_chi_{seat}_{meld_repr}`
- `call_pon_{seat}_{meld_repr}`
- `call_minkan_{seat}_{meld_repr}`

`meld_repr` は構成牌のソート済み連結（`_` 区切り）。

### 5.7 gang（暗槓/加槓）

- `kan_ankan_{seat}_{tile}`
- `kan_kakan_{seat}_{tile}`

`kakan` の場合、槍槓可能な他家へ `opt_react_{seat}_ron` を発行する。

### 5.8 kaigang（新ドラ）

- `dora_{tile}`

### 5.9 hule（和了）

- ロン: `win_ron_{winner}_from_{loser}`
- ツモ: `win_tsumo_{winner}`
- 収支（全員分）: `score_delta_{seat}_{delta}`

### 5.10 pingju（流局）

- `draw_{reason}`（例: `draw_ryukyoku`）
- 収支（全員分）: `score_delta_{seat}_{delta}`

### 5.11 未知イベント

- `event_unknown_{key}`

## 6. 意思決定トークンの設計意図

この tokenizer の中核は次の対を明示すること:

- 候補提示: `opt_self_*`, `opt_react_*`
- 採否結果: `take_*`, `pass_*`

これにより、牌譜に現れない「合法だが未実行」の情報を教師信号として扱える。

## 7. 命名と旧設計との差分

旧設計との主な差分:

- `kaiju/jieju` -> `game_start/game_end`
- `zimo` -> `draw` / `gang_draw`
- `dapai/lizhi` -> `discard` / `riichi`
- `baopai` -> `dora`
- `fenpei` -> `score_delta`
- `hule_*` 詳細（役/符/翻/手牌列）は現状未出力

本リポジトリでは README と本書を正とし、旧命名は参考情報としてのみ扱う。
