## 1. 実装で重要なポイント（要約）

### 1-1. 「選択した行動しかログに出ない」

- 牌譜には実行行動は出るが、未実行の合法手（鳴けたがスルー、リーチ可能だがダマ等）は基本出ない。
- 学習用には、牌譜再生で `legal_actions` を計算して、`pass` 系トークンを補う必要がある。

### 1-2. 牌譜イベントは順序依存

- 局内イベントは時系列で処理しないと状態が崩れる。
- 特に槓まわり（`GANG` / `GANGZIMO` / `DORA`）は、ドラ公開タイミングが実装上の落とし穴になりやすい。

### 1-3. 副露/槓コード（`m`）はデコード前提

- `N` タグの `m` はビット符号化。
- `chii / pon / daiminkan / kakan / ankan` で解釈分岐が必要。
- `m` の扱いを雑にすると、手牌更新がズレて後段で不整合になる。

### 1-4. 赤5・記号付き打牌の扱い

- 赤5は通常5に正規化して扱うか、赤情報を別特徴として保持するかを先に決める。
- 打牌トークンの記号（例: リーチ宣言やツモ切りを示す記号）は意味を落とさず保持する。

## 2. 実装チェックリスト

- `qipai` で全員手牌・点数・ドラ表示・局情報を初期化
- `zimo` / `dapai` ごとに手牌を更新
- `dapai` 直後に他家の反応可能手を計算（`ron/chi/pon/kan`）
- `fulou`/`gang` は `m` デコード結果で消費牌を決定
- 槓後の嶺上ツモとドラ公開順をイベント順で反映
- `hule` / `pingju` で局収支反映
- 学習用出力では `take_*` と `pass_*` を同じ意思決定点で対にする

## 4. このリポジトリでの使い方

- 実装参照先:
  - `src/tenhou_tokenizer/engine.py`
- まずこのノートのチェックリストを満たすようにイベント処理順を固定する。
- 次に、合法手計算を `shape` 判定から `yaku/furiten` 完全判定へ拡張する。
